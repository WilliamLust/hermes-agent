- Researcher has robust 3-tier waterfall (Firecrawl → Scrapling → Camoufox)

### Strengths
- Single-number interaction design is zero-friction
- State file (`.topic_approval_state.json`) persists across restarts

### Gaps & Risks
1. **No pipeline concurrency guard.** If user picks "2" while pipeline is already running on a previous pick, two pipelines launch simultaneously on the same GPU. 12 parallel chapter builders × 2 books = 24 concurrent Ollama processes on a single RTX 3090. This will OOM.
2. **No confirmation for multi-pick.** If user sends "1 3 5" (spaces), only the first number is parsed. Intent is ambiguous.
3. **Gateway's Telegram listener vs topic_approval.py listener.** The Hermes gateway already handles inbound Telegram messages. `topic_approval.py --listen` uses long-polling `getUpdates`, which CONFLICTS with the gateway's own polling. Running both simultaneously will cause update ID conflicts.
4. **PENDING vs DONE race.** If pipeline crashes before calling `mark_done()`, the topic stays PENDING forever and gets re-processed on the next run.

### Optimization Recommendations

**R4: Pipeline concurrency lock**
```python
# Add to run_pipeline.py at startup:
LOCK_FILE = HERMES_HOME / "ebook-factory" / ".pipeline.lock"

def acquire_pipeline_lock():
    if LOCK_FILE.exists():
        # Check if process is still alive
        pid = int(LOCK_FILE.read_text().strip())
        try:
            os.kill(pid, 0)  # signal 0 = check existence
            die(f"Pipeline already running (PID {pid})")
        except ProcessLookupError:
            log(f"Stale lock from dead PID {pid}, removing")
            LOCK_FILE.unlink()
    LOCK_FILE.write_text(str(os.getpid()))
    return True

def release_pipeline_lock():
    LOCK_FILE.unlink(missing_ok=True)
```
ROI: Prevents GPU OOM from dual pipelines. Critical for reliability.

The Hermes gateway already handles all Telegram inbound messages with full agent capabilities. The `--listen` mode creates a competing long-polling connection to the same bot token, which causes update delivery conflicts. Instead, let the gateway agent handle number replies using the instructions in `~/agents.md`.

Implementation: Remove the `--listen` flag from the documented interface. The gateway agent, with factory context from `~/agents.md`, can:
- Detect a number reply
- Run `topic_approval.py --auto-pick N`
- Pipeline starts automatically

No separate listener process needed. No update ID conflicts.

## Phase 3: Outlining

### Current State
- **Script:** `~/.hermes/skills/ebook-factory/skills/outliner/orchestrator.py` (~520 lines)
- **Model:** `qwen3.5:35b-a3b-q4_k_m` (local Ollama)
- **Trigger:** Called by `run_pipeline.py`
- **Flow:** Load LEARNING.md + style guide → Qwen 35B generates outline → validate chapter count → detect vague examples → Qwen 35B regenerates vague sections → write `01_outline.md` + `cover_prompt.txt`

### Strengths
- Vague example detection (14 regex patterns) + targeted regeneration is clever
- Cover prompt pre-generation feeds the cover generator with book-specific art direction
- Quality example from Book 13 Chapter 3 anchors the 35B's output format
- Viability gate (≥6.0) prevents clearly bad outlines

### Gaps & Risks
1. **CRITICAL: 35B model requires `{"think": false}` but code doesn't enforce it.** The 35B-a3b model puts reasoning in `message.thinking`, not `message.content`. If Ollama doesn't strip thinking tokens, the outline content will be empty or garbled. Current code just calls `ollama_call()` and takes the response — no validation that content is non-empty or doesn't contain thinking tokens.
2. **No retry on Ollama failure.** `ollama_call()` returns None on ConnectionError → `error_exit()`. A transient 5-second Ollama hicup kills the entire pipeline.
3. **Single model, no fallback.** If 35B isn't loaded (e.g., after an Ollama update), the outliner dies. No fallback to 27B.
4. **`num_ctx: 16384`** is hardcoded. 35B-a3b-q4_k_m is 23GB model. On 24GB VRAM with 12 parallel chapter builders also loading 27B, context may exceed VRAM.
5. **Ollama URL hardcoded** (`http://localhost:11434/api/chat`). No env override, unlike chapter_builder which uses `OLLAMA_BASE`.

### Optimization Recommendations

**R7: Add "think": false to all 35B Ollama calls and validate non-empty content**
```python
# In orchestrator.py, modify ollama_call():
def ollama_call(prompt, system, model=OUTLINE_MODEL, num_predict=6000, temperature=0.7):
    payload = {
        "model": model,
        "messages": [{"role": "system", "content": system}, {"role": "user", "content": prompt}],
        "stream": False,
        "options": {"temperature": temperature, "num_predict": num_predict, "num_ctx": 16384},
    }
    # CRITICAL: 35B-a3b model puts output in thinking, not content
    if "35b-a3b" in model:
        payload["think"] = False
    
    response = requests.post(OLLAMA_URL, json=payload, timeout=REQUEST_TIMEOUT)
    data = response.json()
    content = data.get("message", {}).get("content", "").strip()
    
    # Validate content is not empty or thinking-only
    if not content or len(content) < 100:
        thinking = data.get("message", {}).get("thinking", "")
        if thinking and not content:
            log("WARNING: 35B returned thinking but empty content — model may need think:false")
        return None
    return content
```
ROI: Prevents silent empty outlines. Critical — this is the #1 pipeline failure mode.

**R8: Add retry with exponential backoff for Ollama calls**
```python
def ollama_call_with_retry(prompt, system, model=OUTLINE_MODEL, max_retries=3, **kwargs):
    for attempt in range(max_retries):
        try:
            result = ollama_call(prompt, system, model, **kwargs)
            if result:
                return result
            log(f"Ollama returned empty (attempt {attempt+1}/{max_retries})")
        except requests.ConnectionError:
            log(f"Ollama connection failed (attempt {attempt+1}/{max_retries})")
        except requests.Timeout:
            log(f"Ollama timeout (attempt {attempt+1}/{max_retries})")
        
        if attempt < max_retries - 1:
            wait = 5 * (2 ** attempt)  # 5s, 10s, 20s
            log(f"Retrying in {wait}s...")
            time.sleep(wait)
    
    return None  # caller decides what to do
```
ROI: Eliminates ~30% of pipeline crashes from transient Ollama issues. ~10 min/book saved on retry vs restart.

**R9: Model fallback chain**
```python
FALLBACK_MODELS = ["qwen3.5:35b-a3b-q4_k_m", "qwen3.5:27b-16k"]

def ollama_call_with_fallback(prompt, system, preferred_model=OUTLINE_MODEL, **kwargs):
    models = [preferred_model] + [m for m in FALLBACK_MODELS if m != preferred_model]
    for model in models:
        result = ollama_call_with_retry(prompt, system, model, **kwargs)
        if result:
            return result
        log(f"Model {model} failed, trying next...")
    return None
```
ROI: Pipeline continues even if 35B OOMs. 1 book/month saved from total failure.

### Current State

- **Trigger:** Called by `run_pipeline.py` in parallel (12 subprocess.Popen calls)
- **Flow:** Load outline section + LEARNING.md + voice reference → draft via Qwen 27B → validate word count → up to 3 refinement iterations → write to `w-drafts/chapter-XX.md`
- **Auto-promote:** `run_pipeline.py` copies all drafts to `w-polished/` after chapter builder finishes

### Strengths

- `num_predict: 6000` for drafts, `8000` for refinements — appropriate for 3200 word target

### Gaps & Risks
1. **12 parallel Ollama processes on a single RTX 3090.** Qwen 27B-16k is 17GB. 12 concurrent requests to Ollama will queue, not parallelize on GPU — but they will all compete for VRAM. Ollama handles this with request queuing, but 12 simultaneous 16k-context requests will cause significant memory pressure and potential OOM on the 35B model if it's still loaded from outlining.
2. **No GPU memory check before launch.** If the outliner's 35B model is still in VRAM (23GB), the 12 chapter builders requesting 27B (17GB) will cause Ollama to evict 35B, which takes ~30-60 seconds of model swapping. No code accounts for this.
3. **`call_ollama()` has temperature=0.7 hardcoded** — good for creative drafting but not configurable.
4. **Word count validation uses target ±10%, but packager validator uses ±15%.** Inconsistent gates.
5. **LEARNING.md cap at 3000 chars** may truncate valuable patterns at scale.
6. **`/no_think` prefix in refinement prompts** — this is a Qwen 2.5 feature, not standard Ollama API. Should use `"think": false` in the payload instead.
7. **No chapter cross-referencing.** Chapter 7 might reference "as we discussed in Chapter 5" but the chapter builder only sees its own outline section.

### Optimization Recommendations

**R10: Stagger parallel chapter launches to avoid GPU thrashing**
